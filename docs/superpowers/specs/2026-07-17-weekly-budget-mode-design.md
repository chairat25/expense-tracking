# โหมดงบรายสัปดาห์ (Weekly Budget Mode) — Design

**วันที่:** 2026-07-17
**สถานะ:** อนุมัติแล้ว

## ปัญหา

สูตรเงินเฉลี่ยต่อวันปัจจุบัน (`src/app/page.tsx:101-116`) คือ:

```
เงินเฉลี่ยต่อวัน = (เงินคงเหลือ ณ ก่อนวันนี้) ÷ (จำนวนวันที่เหลือในเดือน)
```

มันหารใหม่ทุกวันโดยอิงทั้งเดือน ทำให้ **ยิ่งประหยัด ตัวเลขยิ่งพอง**:

- เดือนละ 6,500 ÷ 31 วัน = 210/วัน
- ประหยัด 3 สัปดาห์ ใช้แค่วันละ 100 → วันที่ 22 เหลือ 4,400 ÷ 10 วัน = **440/วัน**
- 440 × 7 = 3,080/สัปดาห์ ทั้งที่ผู้ใช้ได้จริงแค่ 1,500/สัปดาห์

ผู้ใช้เลยเห็นตัวเลขที่ชวนให้ใช้เงินเกินความเป็นจริง

## เป้าหมาย

เพิ่มตัวเลือกให้คำนวณเงินเฉลี่ยต่อวันจาก **งบก้อนรายสัปดาห์** แทนก้อนรายเดือน โดยยังใส่เงินตั้งต้นเป็นรายเดือนเหมือนเดิม

## ขอบเขต (Scope)

**ทำ:**
- โหมดคำนวณ `month` (เดิม) / `week` (ใหม่) สลับได้
- setting ระดับผู้ใช้ ใช้ร่วมกันทุกเดือน
- ปุ่มแก้ไขงบเดิม ปรับให้ apply ทั้งสัปดาห์ได้

**ไม่ทำ (อนาคต):**
- หน้า admin / dashboard สำหรับเปิด-ปิด feature ให้ user จากหลังบ้าน — ตาราง `user_settings` ที่สร้างในงานนี้ออกแบบให้เพิ่มคอลัมน์ flag ต่อยอดได้ แต่ **ไม่สร้างอะไรล่วงหน้าในงานนี้**
- หน้าสรุปรายสัปดาห์แยก (โชว์รายรับ-รายจ่ายรวมต่อสัปดาห์) — งานนี้แตะแค่ตัวเลขเฉลี่ยต่อวัน

## ข้อกำหนดร่วม (Global Constraints)

- **เวลาไทยทั้งหมด** — `src/lib/shared.ts` ล็อก `TZ = "Asia/Bangkok"` ไว้แล้ว (Vercel รันบน UTC) ห้ามเรียก `new Date()` ดิบๆ เพื่ออ้าง "ตอนนี้" — ใช้ `todayKey()` เท่านั้น
- ตัวช่วยเรื่องวันใหม่ทั้งหมดต้องเป็น **การบวกลบปฏิทินล้วน** ตามแพทเทิร์นของ `shiftDate` / `formatDayTH` ที่มีอยู่ (`new Date(y, m-1, d)` แล้วอ่าน `getDay()` — สร้างกับอ่านอยู่โซนเดียวกัน ผลจึงเท่ากันทุกเครื่อง)
- สัปดาห์เริ่ม **วันจันทร์** จบ **วันอาทิตย์** (ISO)
- `db:push` ใช้ไม่ได้ในโปรเจกต์นี้ (drizzle-kit bug ตอน introspect Supabase) — ใช้ `npm run db:generate` แล้ว apply ด้วย `psql "$DATABASE_URL" -f <file>` แทน
- ค่าเงินใน DB เป็น `numeric` ซึ่ง drizzle คืนมาเป็น **string** ต้อง `Number()` ตอนอ่าน และ `.toFixed(2)` / `String()` ตอนเขียน

## สูตรคำนวณ

### โหมด `month` (เดิม ไม่เปลี่ยน)

```
remainingBefore = openingBalance + (รายรับก่อนวันนี้ทั้งเดือน) − (รายจ่ายก่อนวันนี้ทั้งเดือน)
daysLeft        = จำนวนวันที่เหลือในเดือน (นับวันนี้ด้วย)
เงินเฉลี่ยต่อวัน  = remainingBefore ÷ daysLeft
```

### โหมด `week` (ใหม่)

```
weekSlice   = ช่วง [จันทร์..อาทิตย์] ของวันที่ดูอยู่ ตัดขอบให้อยู่ในเดือนนั้น
share       = openingBalance × (จำนวนวันใน weekSlice ÷ จำนวนวันทั้งเดือน)

remainingBefore = share + (รายรับใน weekSlice ก่อนวันนี้) − (รายจ่ายใน weekSlice ก่อนวันนี้)
daysLeft        = จำนวนวันที่เหลือใน weekSlice (นับวันนี้ด้วย)
เงินเฉลี่ยต่อวัน  = remainingBefore ÷ daysLeft
```

สูตรขนานกับโหมดเดือนเป๊ะ ต่างแค่ 2 จุด: ใช้ `share` แทนเงินตั้งต้นเต็มก้อน และหน้าต่างเวลาเป็น weekSlice แทนทั้งเดือน

### หน้าตาของ `computeBudget`

```ts
export type BudgetMode = "month" | "week";

export type BudgetInfo = {
  /** เงินเฉลี่ยต่อวันของวันที่ดูอยู่ */
  amount: number;
  /** true = มาจาก daily_budgets ที่ผู้ใช้กรอกเอง (ไม่ได้คำนวณให้) */
  isManual: boolean;
  /** null ในโหมด month — ใช้โชว์บรรทัดบริบทสัปดาห์ */
  week: { from: string; to: string; envelope: number; daysLeft: number } | null;
};

export function computeBudget(
  month: MonthData,
  date: string,
  mode: BudgetMode,
): BudgetInfo;
```

`week` คำนวณเสมอเมื่อ `mode === "week"` ไม่ว่า `isManual` จะเป็นอะไร — บรรทัดบริบทต้องโชว์ทั้งสองกรณี

### คุณสมบัติที่ได้จากสูตรนี้

- **ไม่ทบข้ามสัปดาห์** — สัปดาห์ใหม่คำนวณจาก `share` ก้อนใหม่เสมอ ใช้เกินสัปดาห์ที่แล้วไม่ตามมาหลอน ประหยัดไว้ก็ไม่พองข้ามสัปดาห์ ตัวเลขนิ่งอยู่แถวๆ 210/วันตลอดเดือน
- **รายรับเข้าสัปดาห์ไหน อยู่สัปดาห์นั้น** — ได้ฟรีจากสูตร ไม่ต้องกดอะไร (นับตั้งแต่วันที่เงินเข้าเป็นต้นไป เหมือนโหมดเดือน) และมันรั่วไปสัปดาห์หน้าไม่ได้เพราะสัปดาห์หน้าเริ่มจาก `share` ใหม่
- **สัปดาห์คาบเกี่ยวข้ามเดือนถูกตัดขอบ** — เช่นสัปดาห์สุดท้ายเหลือ 3 วันในเดือนนี้ ได้งบ 3/31 ของเดือน (ไม่ใช่ 7/31) เพราะแอปโหลดข้อมูลทีละเดือน การดึงข้ามเดือนมาคำนวณต้องรื้อ data loading ทั้งหมด — ไม่คุ้ม
- **ยอด "คงเหลือ" บน header ยังบอกความจริงเหมือนเดิม** — โหมดนี้แตะแค่ตัวเลขเฉลี่ยต่อวัน ถ้าใช้เกินจนเดือนพัง header ยังเตือนอยู่ (safety net ไม่หาย)

### การกรอกงบเอง (manual override)

`daily_budgets` ที่ผู้ใช้กรอกเองมาก่อนเสมอทั้งสองโหมด (พฤติกรรมเดิม ไม่เปลี่ยน)

ต่างกันตรงตอน **เขียน**:
- โหมด `month` — เขียนแถวเดียว (วันที่ดูอยู่)
- โหมด `week` — เขียนให้ **วันที่เหลือในสัปดาห์** คือตั้งแต่วันที่ดูอยู่ → วันอาทิตย์ (ตัดขอบเดือน) รวม 1–7 แถว
  - **ไม่เขียนทับวันที่ผ่านไปแล้ว** — ประวัติจะเพี้ยน

## สถาปัตยกรรม

### Data

ตารางใหม่ `user_settings`:

| คอลัมน์ | ชนิด | หมายเหตุ |
|---|---|---|
| `user_id` | `uuid` PK | = `auth.users.id` |
| `budget_mode` | enum `budget_mode` (`month` \| `week`) | default `'month'` |
| `created_at` | `timestamptz` | default now |

- ใช้ `pgEnum` ให้สอดคล้องกับ `txTypeEnum` ที่มีอยู่
- FK + RLS อยู่ใน `drizzle/rls_settings.sql` (ไฟล์ incremental ใหม่ ตามแพทเทิร์น `rls_savings.sql`) — **ไม่แก้ `rls.sql` / `rls_savings.sql` เดิม เพราะรันไปแล้ว รันซ้ำจะ error**
- ตารางนี้คือที่อยู่ตามธรรมชาติของ feature flag ในอนาคต (เพิ่มคอลัมน์เอา) แต่ตอนนี้มีแค่ `budget_mode`

**ไม่มีตารางใหม่สำหรับงบสัปดาห์** — reuse `daily_budgets` เดิม โดยเขียนค่าเดียวกันลงหลายวัน

### API

| Endpoint | เปลี่ยนอะไร |
|---|---|
| `GET /api/months/[ym]` | เพิ่ม `budgetMode` เข้า response |
| `PUT /api/settings` (ใหม่) | `{budgetMode}` → upsert `user_settings` |
| `PUT /api/months/[ym]/daily-budgets` | body เปลี่ยนจาก `{date, amount}` → `{dates: string[], amount}` คืน array |

**ทำไม `budgetMode` ไปอยู่ใน response ของ months:** มันไม่ใช่ข้อมูลของเดือนจริงๆ (ยอมรับว่าแปลกเชิงความหมาย) แต่แลกมาด้วยการไม่ต้อง fetch เพิ่ม และไม่มีจังหวะโหมดกระพริบผิดตอนโหลด — skeleton ที่ทำไว้คลุมให้อยู่แล้ว การแยก `GET /api/settings` ต่างหากจะได้ความสวยเชิงความหมายแต่เสีย UX ซึ่งไม่คุ้ม

**ทำไม `daily-budgets` ต้องรับ array:** โหมดสัปดาห์ต้องเขียนได้ถึง 7 แถวต่อการกดหนึ่งครั้ง ถ้าคงของเดิมต้องยิง 7 request โหมดเดือนส่ง `dates` ยาว 1 — ยิงครั้งเดียวจบทั้งสองโหมด

### UI

segmented control เล็กๆ ในการ์ดงบของหน้ารายวัน (`#tour-daily-budget`) — ตรงที่ตัวเลขอยู่ กดปุ๊บเห็นเปลี่ยนปั๊บ:

```
┌─────────────────────────────────────┐
│ เงินเฉลี่ยต่อวัน      [ เดือน │ สัปดาห์ ] │
│                    ✏️ แก้ไขงบสัปดาห์นี้  │
│  ┌──────────┬────────┬──────────┐   │
│  │ เฉลี่ย/วัน │  ใช้ไป  │   เหลือ   │   │
│  │   214.29 │ 150.00 │    64.29 │   │
│  └──────────┴────────┴──────────┘   │
│  งบสัปดาห์นี้ 1,500 · เหลืออีก 4 วัน     │
└─────────────────────────────────────┘
```

- บรรทัดบริบทสัปดาห์โผล่เฉพาะโหมด `week` — ไม่งั้นตัวเลขเฉลี่ยลอยๆ ไม่รู้มาจากไหน
- `งบสัปดาห์นี้` (`week.envelope`) = `share` + รายรับ**ทั้ง** weekSlice — คือเงินของสัปดาห์นี้ทั้งก้อน
  - **จงใจให้ต่างจากตัวตั้งในสูตรเฉลี่ยต่อวัน** ซึ่งนับรายรับแค่ *ก่อนวันนี้* ตามธรรมเนียมที่ทั้งแอปใช้อยู่ — ไม่ใช่บั๊ก อย่า "แก้" ให้ตรงกัน
  - ผลคือถ้าล่วงหน้าบันทึกรายรับของวันหลังในสัปดาห์เดียวกันไว้ ยอด envelope จะรวมให้แล้วแต่ค่าเฉลี่ยต่อวันยังไม่รวม — ทั้งคู่จะบรรจบกันเองเมื่อถึงวันนั้น
- ปุ่มแก้ไขเปลี่ยนคำตามโหมด: `แก้ไขเงินเฉลี่ยต่อวัน` ↔ `แก้ไขงบสัปดาห์นี้`
- คง `id="tour-daily-budget"` ไว้ (driver.js tour อ้างอยู่)
- ตัวสลับซ่อนตอนเดือนปิดยอด (`locked`) เหมือนปุ่มแก้ไข

## Error handling

- `PUT /api/settings` — zod validate `budgetMode` ต้องเป็น `'month' | 'week'` เท่านั้น ไม่ผ่าน → 400
- `PUT .../daily-budgets` — validate ทุก date ใน `dates[]` ต้องขึ้นต้นด้วย `ym` ที่ส่งมา ไม่ผ่าน → 400 (กันเขียนข้ามเดือน)
- `dates[]` ต้องไม่ว่างและยาวไม่เกิน 7
- ผู้ใช้ที่ยังไม่มีแถวใน `user_settings` → default `'month'` (ไม่ error, ไม่ต้อง seed)
- ฝั่ง client ใช้ `setError()` เดิมที่มีอยู่ ไม่เพิ่มกลไกใหม่

## Testing

โปรเจกต์ยังไม่มี test infra เลย งานนี้ตั้ง **vitest** ขึ้นมาเฉพาะ pure functions ใน `shared.ts` — จุดที่คุ้มที่สุด เพราะคณิตสัปดาห์พังเงียบได้ง่าย (ขอบสัปดาห์ ตัดขอบเดือน เดือนที่ขึ้นต้นวันอาทิตย์ ฯลฯ) ไม่ตั้ง Playwright/E2E ในงานนี้

เคสที่ต้องคลุม:
- `weekStart` — วันจันทร์คืนตัวเอง, วันอาทิตย์ถอย 6 วัน, ข้ามเดือน, ข้ามปี
- `weekSliceInMonth` — สัปดาห์เต็ม 7 วัน, สัปดาห์แรกของเดือนที่ถูกตัด, สัปดาห์สุดท้ายที่ถูกตัด, เดือนที่วันที่ 1 เป็นวันอาทิตย์
- `dayCount` — วันเดียวกัน = 1, ข้ามเดือน, ข้ามปีอธิกสุรทิน
- `computeBudget` โหมด month — ตรงกับพฤติกรรมเดิมเป๊ะ (regression guard)
- `computeBudget` โหมด week — สัปดาห์เต็ม, สัปดาห์ถูกตัดขอบ, มีรายรับกลางสัปดาห์, ใช้เกินงบ (ค่าติดลบ), manual override ชนะ
- **ไม่ทบข้ามสัปดาห์** — ใช้เกินสัปดาห์ที่ 1 แล้วสัปดาห์ที่ 2 ต้องได้งบเต็ม (นี่คือหัวใจของ feature ต้องมี test ตรงๆ)

ทดสอบด้วยมือหลังจากนั้น: สลับโหมดแล้วตัวเลขเปลี่ยน, แก้งบสัปดาห์แล้วทุกวันที่เหลือเปลี่ยนตาม แต่วันที่ผ่านมาไม่เปลี่ยน, สลับโหมดแล้วเปลี่ยนเดือน โหมดยังคงอยู่, refresh แล้วโหมดยังอยู่
